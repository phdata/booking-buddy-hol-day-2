Act 1 · Context Engineering
What is CLAUDE.md?
→
What it is
A markdown file Claude Code reads automatically when invoked in a directory. No prompt needed — it loads on every invocation and is always in scope.
!
The always-on cost
Every byte counts against the context window on every invocation. A 400-line root file means every agent loads docs it doesn't need — degrading output and wasting tokens.
→
Resolution order
Claude reads the innermost file first, then walks outward through parent directories, then user, then enterprise. All matching files load and stack — inner scope wins on conflicts.
→
Multiple scope levels
CLAUDE.md can live at four scopes: enterprise, user, project, and local. Within a project, nest additional files inside service or module directories for targeted context.
✓
The project hierarchy pattern
Keep root and user files to cross-cutting concerns only — CI, git conventions, shared style. Push service-specific rules down into subdirectory files.
Multiple scope levels
Enterprise /etc/claude/CLAUDE.md org-wide policies
User ~/.claude/CLAUDE.md personal prefs
Project ./CLAUDE.md committed, shared
Local ./CLAUDE.local.md gitignored overrides
All files load. Inner scope wins on conflict.
The project hierarchy pattern
CLAUDE.md repo root — cross-cutting rules
booking-service/CLAUDE.md
fleet-service/CLAUDE.md
loyalty-service/CLAUDE.md
notifications-service/CLAUDE.md

Act 2 · Skills
What are Skills?
→
What they are
Markdown files in .claude/skills/ — reusable instruction sets Claude can invoke as part of any task. Write them once; they're available to every session in this repo.
→
How Claude decides to use one
Claude matches the task against each skill's description: and when_to_use: front matter. Those two fields are the entire signal — vague = ignored or misfiring.
!
Failure modes
description: 'helps with things' — never fires (nothing maps to 'things'). description: 'checks code' — fires on everything (all tasks touch code). Both are broken.
✓
Specificity is the design challenge
A good skill fires on exactly the right task and stays silent on everything else. That requires when_to_use: grounded in real repo paths, error patterns, and task keywords.
Core Front Matter
Core Front Matter
name — kebab-case slug; appears in logs and governance tooling
description — must name the task type, not just the domain
when_to_use — specific triggers: file paths, error messages, task keywords
body — grounded in actual service code; generic advice gets ignored
Key Concepts
Front matter Trigger conditions Activation specificity Team governance

Act 3 · MCP
What is MCP?
→
Model Context Protocol
Open standard connecting AI models to external tools and live data. With MCP, Claude queries databases, calls APIs, reads dashboards.
→
Architecture: Host → Client → Server
Claude Code is the Host; it manages an MCP Client that connects to Servers. Each Server is independent; multiple can run simultaneously.
→
Three primitives
Tools: callable functions. Resources: structured data Claude can read. Prompts: reusable templates.
!
Tool sprawl
Similar tools across servers cause misrouting — Claude picks the wrong one. Fewer, focused tools = better selection.
✓
Why the pre-built server
mcp/fleet-status/ ships already written. Focus on integration — connecting, testing, pruning — not authoring.

Act 4 · Trust & Safety
Hooks & Permissions
Hook Lifecycle Events
Hooks are shell scripts that run at specific points in Claude's workflow. Configure them in .claude/settings.json. Non-zero exit code = action blocked.
PreToolUse
Fires before any tool call. Log, validate, or block. Most common hook for audit and enforcement.
PostToolUse
Fires after a tool call completes. Use to log results or trigger side effects.
Stop
Fires before Claude marks a task complete. Gate on test results. Prevent shipping broken code.
Notification
Fires when Claude sends a notification. Use to route alerts to Slack, email, or logging systems.
Permission Model
Two Modes
Allowlist only explicitly permitted actions run; everything else blocked. Strictest — good for prod agents.
Blocklist only explicitly denied actions are blocked; everything else runs. Good for dev with guardrails.
settings.json 
{
  "permissions": {
    "allow": ["Bash(git *)", "Read", "Write"],
    "deny":  ["Bash(rm -rf *)", "Bash(curl *)"]
  }
}
Hooks = soft guardrails. Run scripts, log, block — but require implementation.

Permissions = hard limits. Enforced by Claude Code before hooks even run. Cannot be bypassed.
Key Concepts
PreToolUse Stop Allowlist vs Blocklist Soft vs hard guardrails

Act 5 · Ship It
Parallel Agents & Worktrees
Sequential vs Parallel
Sequential (default)
Task A
Task B — waits for A
Task C — waits for B
~3× longer · context switches between services
Parallel (what you'll do)
Task A
Task B
Task C
All three run simultaneously · each agent has isolated context
Worktrees
Problem: Speculative work (a migration plan, a risky refactor) shouldn't touch your working branch. If it goes wrong, you've polluted the branch everyone else is using.
What a worktree gives you
Isolated copy of the repo on a separate git branch
Agent works there; your main branch is untouched
Merge only if you decide to adopt the work
Use isolation: "worktree" in the dispatch prompt
Task C in the next activity uses a worktree — it's a migration plan. You might adopt it, you might not.
Key Concepts
Parallel dispatch Context isolation Worktrees Speculative work
